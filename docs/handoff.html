00. 앱 쉘 · 로그인 · 권한

사이드바 상단 지역 드롭다운 · 탑바 로그인 핫픽 · 로그인 시트 · 사용자 메뉴(권한)

기획의도 — "우리 마을 사업의 이익은 실제 주민에게 돌아가야 한다"는 전제를 인증 흐름에서부터 가시화. 카카오 로그인(편의) → PASS 본인확인(신원) → 거주지 증빙(자격) 3단계를 모두 통과해야 resident 권한이 부여된다. 운영자 권한은 곡성군청 · 조합 사무국에 한해 부여되며, 프로토타입에서는 토글 버튼으로 전환 가능하다.

🔗API 연결지점

메서드엔드포인트화면 트리거api.js
POST/api/auth/login/kakao로그인 시트 · "카카오로 계속하기"API.auth.kakaoLogin()
POST/api/auth/verify/passPASS 휴대폰 본인확인API.auth.verifyPass()
POST/api/auth/verify/residence주민등록 거주지 증빙API.auth.verifyResidence({region,emd})
GET/api/auth/me앱 부트 / 새로고침API.auth.me()
POST/api/auth/roles/toggle-admin데모 전용 — 운영 시 RBAC 승격으로 대체API.auth.toggleAdminDemo()
GET/api/regions사이드바 지역 드롭다운API.regions.list()

📦핵심 응답 shape (`/api/auth/me`)

{ data: { id, name, kakao_id, phone_masked, email, roles: ['resident','member', 'admin'?], residence: { region_code, region_name, emd_code, emd_name, verified_at }, pass_verified_at, kakao_verified_at, member_no, created_at, } }
운영 시 유의 — `resident` 권한은 거주지 증빙 완료 이후에만 부여. DB 레벨에서 CHECK 제약. admin 은 항상 `scope_region_code` 가 함께 부여되어야 함 (곡성군 admin 이 다른 지역을 관리하지 못하도록).

🧰부가 스펙

  • 지역 전환 — 클라이언트 state 만 갱신(`selectRegion`), 해당 지역의 EMD/후보지/사업/변전소 재조회 필요. ?region= 쿼리 표준.
  • 세션 — JWT(`Authorization: Bearer`) 권장. `credentials: include` 대응 가능.
  • Tweaks(화면 설정) — 글자크기·블루톤·여백밀도는 클라이언트 전용(body data-* 속성). 서버 저장 불필요.

01. 홈 home주민

히어로(인사 + 마스코트) → 3개 진입카드 → 동네 소식 → 최신뉴스

기획의도 — 첫 화면은 "우리 마을 사업이 어디까지 왔는지"를 확인하는 대시보드가 아니라, 주민이 뭘 먼저 해볼 수 있는지를 안내하는 랜딩이다. 세 개의 진입카드 — 자유문의(이해), 우리 조합(참여), 지도(현황) — 는 "어려운 말 → 쉽게 설명" / "남의 일 → 내 조합" / "추상 → 공간" 3 축을 커버. 동네 소식 / 최신뉴스는 플랫폼이 "아직 활동이 살아있다"는 신호용이며, 운영자 직접 등록 또는 자동 수집.

🔗API 연결지점

메서드엔드포인트용도api.js
GET/api/home/feed?region=gokseong동네 소식 (설명회·준비중 사업·참고사례)API.home.feed(region)
GET/api/home/news?region=gokseong최신뉴스 (외부 언론 수집)API.home.news(region)

🧰부가 스펙

  • cached 피드·뉴스는 5~15분 캐시. 자동수집이라 실시간성 낮음.
  • home_feed_items.status — `pending`(예정)/`plan`(준비)/`run`(완료) 3가지로 UI 뱃지 매핑.
  • 뉴스는 외부 URL 링크만 — 본문 저장 금지(저작권). 썸네일은 자체 스크레이핑 또는 OG 이미지.

02. AI · 자유 문의 chat-free주민

퀵 질문 5개 + 자유 입력 · AI 답변 + 출처 칩

기획의도 — "재생에너지는 어렵다"는 진입장벽을 낮추기 위한 설명형 에이전트. 질문 5개(태양광 장단점 · REC 쉬운 설명 · 곡성 히스토리 · 참여 3가지 방식 · 주민수용성)는 퍼널 앞단을 설계하면서 실제로 가장 많이 나올 질문을 프리셋. 답변엔 반드시 출처 칩이 달려 "내 판단을 AI가 대신하는 게 아니라, AI가 나를 돕는다"는 신뢰감을 만든다.

🔗API 연결지점

메서드엔드포인트용도api.js
POST/api/chat/free질문 전송 → AI 답변 (RAG) streaming 권장API.chat.ask(q, region)
Request: { question: "REC가 뭔가요?", region: "gokseong" } Response: { data: { answer_html: "<h4>…</h4>…", sources: ["곡성군 재생에너지 기본계획 (2026.02)", …] } }

🧰부가 스펙

  • 3rd party Claude/OpenAI + documents 테이블 기반 RAG. 문서는 운영자가 `admin-docs` 에서 업로드.
  • answer_html — 서버에서 sanitize 후 전달. 허용 태그: `p, ul, ol, li, strong, em, h3, h4, dl, dt, dd`. script/iframe 금지.
  • sources — 문자열 배열(출처 표시용). 클릭 시 문서 원문 이동이 필요하면 `{label, url, chunk}` 객체로 확장.
  • 로깅 — 모든 질의는 query_logs 에 저장. 유사질문 클러스터링은 배치로 `canonical_question` 채움 → `admin-logs` 에서 활용.

03. AI · 햇빛소득 진단 chat-haebit주민

9단계 진단 플로우 (주소 → 필지 지도 선택 → 토지/규제/일사/계통 자동조회 → 수용성 질문 → 예상 규모 → 최종)

기획의도 — "우리 마을에 태양광 되나요?" 질문을 3분 안에 답하는 셀프 진단. 핵심은 5단계의 자동조회 단계(토지·규제·일사·계통)에서 주민이 별도 자료 없이도 전문가 판단 근거를 얻을 수 있게 하는 것. 마지막 단계는 결과를 예상 용량·사업비·수익으로 환산해 "다음 단계" 콜투액션(조합 가입/에너지팀 상담)으로 연결. 별도 프로덕트로 분리 가능성을 고려해 사이드바에서도 독립 카테고리로 노출.

🔗API 연결지점 — 단계별

🧰부가 스펙

  • 3rd party lookup 계열은 외부 공공데이터 통합 — VWORLD(지목), 농지은행, 임야정보, 기상청 일사량, 한전 계통정보. 초기엔 곡성군 사전계산 CSV fixture 로 시작해도 OK.
  • parcels — 현재 `MOCK_PARCELS` 7개는 1필지(A~G) SVG 도형. 실제 구현 시 WKT/GeoJSON 반환해 Leaflet 에 그림. 주소 → 인접 필지 lookup 알고리즘 필요.
  • 산식은 클라이언트 추정치 (면적 10㎡=1kW, 1MW=14억원=연 2.2억 수익). 정밀 계산이 필요해지면 서버 `finalize` 로 이동 권장.
  • 세션 저장 — 단계별 PATCH 로 incremental 저장. 사용자가 중간에 이탈해도 이어하기 가능해야 함 (기획상 고려, UI 미구현).
  • 결과 CTA — 마지막 화면의 "협동조합 가입하기" 버튼이 실제 가입 플로우로 이어지려면 `coop/applications` POST 와 연결.

04. 지역 대시보드 (지도) dashboard주민

Leaflet 지도 (ESRI satellite) · 4 레이어 · 좌측 사이드(프로필 + 레이어 토글 + 상위 5 면 + 주목 후보지) · 클릭 시 상세 drawer

기획의도 — "우리 군에 어디가 사업에 적합한가"를 한 장의 지도로 설명. 11개 법정면을 색상 온도(0-100 적합도)로 칠하고, 클릭하면 인구·계통·제약·진행사업까지 한 번에 본다. 부지 후보지는 별 핀 + 점수로 강조 — 주민이 "우리 동네에 이렇게 후보가 있다"를 체감하게 한다. 적합도 산식 = 일사 25% + 계통 30% + 부지 25% + 수용성 20% (app.js GS_EMDS.forEach).

🔗API 연결지점

메서드엔드포인트용도api.js
GET/api/regions/:code/emds법정면 11개 + 지표 4종 + suitabilityAPI.regions.emds(code)
GET/api/regions/:code/candidates후보 부지 8곳API.regions.candidates(code)

GET/api/regions/:code/geojson법정면 폴리곤 (EPSG:5186→WGS84) staticAPI.regions.geojson(code)

🧰부가 스펙

  • suitability 산식은 DB GENERATED 컬럼 또는 뷰로 처리 권장. 지표 업데이트 시 자동 재계산.
  • `grid_residual_mw` 는 EMD → nearest_sub 매핑에서 파생 캐시. 변전소 업데이트 시 전파 배치 필요.
  • 지도 폴리곤 — 현재 `GS_EMDS[].poly`(4점 사각형)은 프로토 fallback. 실제로는 `window.GOKSEONG_EMD_GEOJSON`(LSMD shapefile 변환)이 있으면 우선 사용. 배포 시 `/api/regions/:code/geojson` 정적 파일 CDN.
  • 호버 카드(md-hover) — 클라이언트 렌더링. 서버에서 별도 데이터 불필요.
  • 접근성 — Leaflet 클릭 전용이므로 키보드 내비게이션은 좌측 "상위 5개 면 / 주목 후보지" 리스트로 대체.
더미 vs 실제 — 지표 4종(solar/grid/land/accept)은 현재 수작업 점수. 실 배포 단계에서는 ① 기상청 일사량, ② 한전 KEPCO API (계통잔여), ③ VWORLD 토지이용 통계, ④ 설문·조합원 밀도 기반 계산기 로 교체 필요. 교체 전까지는 주기적(월 1회) 수동 업데이트로 운영.

05. 내 조합 coop주민

내 자산(출자금/펀드/지급받음) → 배당 차트 + 수령 내역표 → 내 참여 사업 진행률 → 공지/총회 + 블록체인 활동 기록

기획의도 — "돈이 어디서 얼마나 들어왔고, 어디로 가고 있는가"를 조합원 본인 관점으로 보여주는 페이지. 배당이 곡성사랑상품권으로 전액 수령됨을 반복 노출해 지역 환류 구조를 학습시킴. 하단 "블록체인 원장 기반 · 변경 불가" 카피는 신뢰성 축이므로 기능 구현 여부와 별개로 UX상 필수.

🔗API 연결지점

메서드엔드포인트용도api.js
GET/api/coop/me히어로 + 3카드 (출자금/펀드/지급받음)API.coop.me()

🧰부가 스펙

  • `coop/me` 는 aggregate 엔드포인트. 내부적으로 coop_members + coop_contributions + fund_investments + dividends 합산.
  • `preferred_method: 'VOUCHER'` 가 응답에 있으면 UI에 "전액 곡성사랑상품권"으로 표기. 지역별로 꺼져있으면 자동 `CASH` fallback.
  • 활동 로그는 append-only. `tx_hash` 는 블록체인 연동 전엔 null, 연동 후 채움.
  • realtime 배당 흐름 차트는 월별 aggregate. 캐시 5분 OK.

06. 조합원 명부 coop-members주민/조합

명부 테이블(검색+거리/상태 필터+페이지네이션) · 승인 대기 카드 · 출자금/배당 요약

기획의도 — 조합원이 다른 조합원을 볼 수 있어야 조합이 실재한다는 감각이 생긴다. 다만 개인정보 보호를 위해 이름·생년·주소(동리 단위)·거리구분·출자금·전년 배당만 노출 — 전화번호·계좌·상세주소는 뺀다. "거리 구분(ADJACENT/NEARBY/GENERAL)"은 발전소 기준 500m/1km로 이익 분배 비율을 정하는 핵심 필드.

🔗API 연결지점

메서드엔드포인트용도api.js
GET/api/coop/members?q=&distance=&status=&page=명부 (필터·페이지) + summary metaAPI.coop.members(q,filters)

GET/api/coop/members/applications승인 대기 3건API.coop.pendingApps()

🧰부가 스펙

  • 엑셀 내보내기 — `feature_flags.excel_export_enabled` 가 ON 일 때만 버튼 노출. 서버 generate 후 presigned URL.
  • 이름/주소 마스킹 — 현재 프로토는 풀 노출. 실 배포 시 기본 부분 마스킹으로 전환, "상세" 버튼에서만 풀노출(권한 체크).
  • 필터 — `distance` ∈ {ADJACENT, NEARBY, GENERAL}, `status` ∈ {ACTIVE, PENDING, SUSPENDED, WITHDRAWN}.
  • 승인은 2인 승인이 필요할 수 있음 (정관 확인 필요). 현재 프로토는 1회 클릭으로 `PENDING→ACTIVE`.

07. 총회 · 의결 coop-vote조합원

진행 중 투표 카드(정족수+찬반기권 실시간) · 다음 안건 4건 · 과거 의결 기록

기획의도 — 정기총회 현장에 오지 못해도 조합원이 전자 투표로 의사결정에 참여할 수 있게. 투표 결과는 블록체인 원장에 해시로 기록 — 사후 분쟁/감사 시 변경 여부를 검증할 수 있음. "24시간 이내 1회 변경 가능" 규칙은 실수한 투표를 되돌릴 여유를 주되 무한 변경으로 혼란이 생기지 않도록 한 장치.

🔗API 연결지점

메서드엔드포인트용도api.js
GET/api/coop/votes/current진행중 안건 · 정족수 · 실시간 집계 realtimeAPI.coop.votes.current()
GET/api/coop/votes/agenda이번 총회 전체 안건 4건API.coop.votes.agenda()
GET/api/coop/votes/history과거 의결 기록API.coop.votes.history()
POST/api/coop/votes/:id/ballot투표 / 변경 writeledgerAPI.coop.votes.cast(id, choice)

🧰부가 스펙

  • 상태머신 — votes.status: `PENDING(예정)` → `ACTIVE(진행중)` → `PASSED/REJECTED/EXPIRED`.
  • 정족수 — `quorum_required` = 안건 시작 시점의 ACTIVE 조합원 수 스냅샷. 이후 가입자는 영향 없음.
  • 변경 창 — `vote_ballots.changeable_until` = `cast_at + 24h` 또는 `deadline_at` 중 이른 쪽. 이 시각 이전에만 POST 재호출 허용, 이후는 409.
  • 실시간 집계 — 폴링 10초 / WebSocket. 정족수 도달 시 UI에 "정족수 충족" 뱃지 즉시 갱신.
  • my_choice 필드로 "내 투표" 하이라이트.

08. 운영 대시보드 admin-home운영자

4개 KPI 카드(MAU/인증완료/일평균 AI 질의/햇빛 진단 완료) · 주간 질의 추이 차트 · 상위 질문 카테고리

기획의도 — 곡성군청 에너지팀 주무관이 주 1회 열어 보기 좋은 형태로. KPI는 목표 대비 달성률이 한눈에 보이도록 (500 목표 → 612 달성 ✓ 같은 형태). 상위 질문 카테고리는 다음 주 해야 할 일(FAQ 승격·용어 콘텐츠 보강) 힌트 역할.

🔗API 연결지점

메서드엔드포인트용도api.js
GET/api/admin/kpis?region=4개 KPI + 주간 추이 7일API.admin.kpis(region)
GET/api/admin/queries/top상위 질문 4건 (유사질문 클러스터)API.admin.topQueries()

🧰부가 스펙

  • KPI는 app_kpi_daily 집계 테이블에서 조회. 야간 배치(00:30)에 전일자 기록.
  • 목표(goal)는 설정값 — `region_kpi_goals` 류 테이블이 필요하거나 설정 YAML.
  • "보고서 다운" 버튼 — `/api/admin/reports/monthly` 로 PDF/Excel 생성. 구현 시기 미정.

09. 운영 · 문서 관리 admin-docs운영자

드래그&드롭 업로드 박스 · 최근 문서 리스트 (청크 수·질의 횟수·L0~L3 공개범위)

기획의도 — AI 답변의 품질은 RAG 문서 품질로 결정된다는 전제에서, 운영자가 직접 문서를 관리하는 화면. 공개 범위 `L0~L3` 는 문서가 어느 권한까지 노출되는지 통제 — 내부 회의록(L3)이 주민 질문에 인용되면 안 된다.

🔗API 연결지점

메서드엔드포인트용도api.js
GET/api/admin/docs문서 리스트 + 메타API.admin.docs.list()
POST/api/admin/docs (multipart)업로드 → 비동기 색인 writeAPI.admin.docs.upload(file)
PATCH/api/admin/docs/:id/visibility공개 범위 변경API.admin.docs.setVisibility(id,level)

🧰부가 스펙

  • 파일 타입 — PDF, HWP, DOCX, XLSX. HWP 파싱은 외부 라이브러리/서비스 필요 (한글과컴퓨터 Converter API 등).
  • 색인 상태 — `documents.status`: `uploaded → indexing → indexed`(또는 failed). 업로드 직후엔 `indexing` 으로 리스트에 표시되어야 함.
  • 공개 범위 — `L0`(공개) / `L1`(회원) / `L2`(주민전용) / `L3`(어드민). RAG 질의 시 사용자 권한 기준으로 필터.
  • questions_hit — 해당 문서가 `chat_messages.sources` 에 인용된 횟수. 배치로 카운트 업데이트.

10. 운영 · 질의 로그 admin-logs운영자

최근 7일 질의 로그 · 유사질문 수 · 만족도 · "FAQ로 승격" 일괄 액션

기획의도 — 자주 나오는 질문을 FAQ로 승격해 답변 일관성을 높이고, 운영자가 AI 답변 품질을 간헐적 샘플링 없이 통계적으로 파악할 수 있도록. 만족도는 5점 척도(답변 하단 👍/👎 집계), 일관성 점검 플래그는 "같은 질문인데 답변이 다르다"는 케이스 감지용.

🔗API 연결지점

메서드엔드포인트용도api.js

POST/api/admin/queries/:id/promoteFAQ 승격 writeAPI.admin.promoteToFaq(id,body)

🧰부가 스펙

  • 클러스터링 — `query_logs.canonical_question` 배치 채움 (임베딩 유사도 > 0.85 같은 기준). 리스트는 canonical 기준 group by.
  • FAQ 승격 — `faqs` 레코드 생성 + `query_logs.promoted_from_query_id` 링크. 승격 후 동일 canonical 질의는 FAQ를 우선 인용.

11. 운영 · 조합 운영 admin-coop운영자

4 KPI · 배당 집행 파이프라인 (4단계 상태머신) · 명부 변경 로그 · 상품권 집행 준비 · 기능 플래그

기획의도 — 이 화면이 플랫폼 가치의 핵심. 기존에 엑셀로 하던 배당 계산 · 승인 · 이체 프로세스를 앱 안에서 감사 가능한 상태머신으로 처리. 2인 승인(조합장 + 감사)이 파이프라인의 안전장치. 지역화폐 API 자동 연동으로 현금 이체 대신 곡성사랑상품권 일괄 발행 — 주민에게는 +3% 가산 혜택, 지역에는 환류 효과.

🔀상태머신 — 배당 집행 파이프라인

DRAFT(계산 완료) ↓ 자동: 월초 배치에서 period_end 도달 시 REVIEW(조합장 검토) ↓ POST /api/admin/coop/dividends/:id/approve (chair 역할 필요) APPROVED(감사 서명 대기) ↓ POST /api/admin/coop/dividends/:id/audit-sign (auditor 역할 필요) PAID(이체 파일 생성 + 상품권 API 일괄 발행)

🔗API 연결지점

메서드엔드포인트용도api.js
GET/api/admin/coop/dividends/current진행중 period + 승인 상태API.admin.dividend.pipeline()
POST/api/admin/coop/dividends/:id/approve조합장 승인 writeAPI.admin.dividend.approve(id)
POST/api/admin/coop/dividends/:id/audit-sign감사 서명 writeAPI.admin.dividend.auditSign(id)
GET/api/admin/coop/dividends/:id/transfer-file이체 파일 미리보기(엑셀)API.admin.dividend.transferFile(id)
POST/api/admin/coop/dividends/:id/notify조합원에게 안내 발송API.admin.dividend.notifyMembers(id)
GET/api/admin/coop/member-logs명부 변경 로그 ledgerAPI.admin.memberLogs()
GET/api/admin/voucher/batch/current상품권 배치 준비 현황 외부 APIAPI.admin.voucherBatch()

PUT/api/admin/flags/:keyon/off writeAPI.admin.flags.set(key,enabled)

🧰부가 스펙

  • 2인 승인 제약 — DB CHECK: `state='PAID'` 는 `chair_approved_at AND auditor_signed_at` 모두 NOT NULL 이어야 함. 애플리케이션 레벨에서도 중복 체크.
  • 권한 분리 — `chair` 와 `auditor` 는 서로 다른 user 여야 함 (RBAC 체크 필수). 현재 프로토에는 미구현.
  • 이체 파일 — 전자금융 표준 포맷(ERP 연동용 Excel) 생성. 정확한 스펙은 조합이 사용하는 은행 업체에 따라 다름 — 곡성군민에너지협동조합 기준 확인 필요.
  • 지역화폐 API — 곡성사랑상품권 발행 외부 API 연동. `feature_flags.voucher_api_enabled` OFF 시 자동 CASH fallback.
  • 기능 플래그 — `feature_flags` 테이블 (coop_id 단위). 코드 배포 없이 on/off 가능. UI 카드는 PUT 즉시 반영.
  • 명부 변경 로그 — `activity_logs` 중 `actor_kind='admin'` + member 관련 action 필터링. 해시 `tx_hash` 는 블록체인 연동 후 채움.
외부 의존성 체크리스트 — ① 은행 ERP 업무양식(이체 파일) · ② 곡성군 지역화폐 API 명세 · ③ 블록체인 원장(가칭 프라이빗 체인 or Hyperledger) · ④ 감사 서명 도구(전자서명 인증서 or 회계법인 API). 현 프로토타입에서는 모두 모킹.
